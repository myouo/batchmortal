# 项目总结：Batch Mortal

## 项目概述

`Batch Mortal` 是一个围绕雀魂牌谱批量分析构建的自动化工具。它基于 Python 和 SeleniumBase，从 `amae-koromo` 拉取玩家对局记录，再把牌谱自动提交到 Mortal 分析站点 `mjai.ekyu.moe`，等待分析完成后，将结果导出为 `xlsx` 或 `csv`，并可进一步生成图表报告。

它不是一个单纯的网页抓取脚本，而是一条完整的自动化流水线，覆盖了：

- 获取玩家记录
- 构造牌谱 URL
- 自动提交分析
- 等待并提取结果
- 持久化保存结果
- 生成可视化报告

从结构上看，项目已经具备清晰的模块分工：

- [main.py](D:/projs/personalprojects/batchmortal/main.py)：主入口，负责参数解析、任务组织和流程调度
- [batchmortal/api.py](D:/projs/personalprojects/batchmortal/batchmortal/api.py)：负责玩家搜索、记录拉取和牌谱 URL 构造
- [batchmortal/browser.py](D:/projs/personalprojects/batchmortal/batchmortal/browser.py)：负责浏览器自动化、提交协调、失败恢复
- [batchmortal/results.py](D:/projs/personalprojects/batchmortal/batchmortal/results.py)：负责结果写入、去重和已处理任务识别
- [batchmortal/visualize.py](D:/projs/personalprojects/batchmortal/batchmortal/visualize.py)：负责图表和报告生成

## 项目亮点

### 1. 做通了完整链路

这个项目最明显的亮点，不是某个单独功能，而是把一整条原本需要手工操作的流程彻底打通了：

1. 查玩家
2. 拉牌谱
3. 打开分析网站
4. 提交分析
5. 等待结果
6. 保存结果
7. 输出报表

这意味着它已经从“脚本”跨到了“工具”的层面。

### 2. 重点解决了真实难点

自动化项目真正难的部分往往不是提交表单，而是和外部不稳定因素对抗。这个项目把精力集中在了最现实的问题上：

- Cloudflare Turnstile 验证
- 提交节奏控制
- 浏览器页面失效
- token 超时与拒绝
- 远端站点限流
- 重试与冷却策略

这些逻辑在 [batchmortal/browser.py](D:/projs/personalprojects/batchmortal/batchmortal/browser.py) 里占据了核心位置，说明项目不是停留在“理论可行”，而是已经被真实环境反复锤过。

### 3. 有明显的产品化意识

项目后期已经不只是“作者自己能跑”：

- 支持昵称和账号 ID 两种输入
- 支持配置文件运行
- 支持代理自动检测
- 支持截图保存
- 支持结果图表生成
- 支持自动安装 `uc_driver`

这些设计都在降低使用门槛，提高重复使用的便利性。

### 4. 结果沉淀做得比较完整

项目没有停留在“页面上看一眼结果”，而是把分析结果沉淀成结构化数据，包含：

- 牌谱链接
- 模式
- 开始结束时间
- rating
- AI 一致率
- 温度
- 对局长度
- 玩家 ID
- 审查耗时
- 结果截图路径

这使得它不仅能“分析一次”，还能支持后续复盘、归档和趋势观察。

## 项目中的设计考虑

### 1. 稳定性优先，而不是盲目追求速度

从提交历史和当前代码都能看出，作者曾尝试过更激进的并发方案，但最终主干思路已经偏向“受控提交”和“稳定优先”。

README 中明确区分了：

- 默认模式：单窗口串行，最稳
- `--prewarm-standby`：实验性的双窗口轮转
- `--unsafe-parallel-review`：高风险并发，不推荐

这说明作者已经认识到，性能瓶颈不在本地程序本身，而主要在远端站点、Cloudflare 和代理环境上。

### 2. 认真考虑了重复执行成本

项目不是“一次性跑完就结束”的设计，而是考虑了批量场景和重跑成本：

- 读取已有结果文件
- 根据 UUID 识别已完成任务
- 跳过已成功处理的牌谱
- 避免重复写入

这在批量分析时非常重要，可以显著减少浪费的请求、浏览器操作和等待时间。

### 3. 考虑了部署与使用门槛

项目提供了配置模板 [config.default.yaml](D:/projs/personalprojects/batchmortal/config.default.yaml)，并支持配置文件覆盖命令行参数。对于经常重复使用的人来说，这比每次手敲一长串参数更实用。

同时，项目还加入了 `uc_driver` 自动安装逻辑，减少了 SeleniumBase 环境准备的摩擦。

### 4. 考虑了结果可读性与后续利用

项目不仅导出表格，还支持生成 HTML/PNG 图表报告，把原始数据转化为直观的趋势观察结果。这说明作者关注的不只是“抓到结果”，而是“让结果有可读性、有复盘价值”。

## 项目的意义

### 1. 它把重复、繁琐、低效的人工流程自动化了

原本用户需要手工查玩家、复制牌谱链接、打开分析网站、等待结果、再手动记录。这个项目把这些重复劳动整合成了一次性批处理流程，大幅降低了重复操作成本。

### 2. 它让牌谱分析从“单次查询”变成“结构化积累”

如果只是网页上临时看结果，价值是瞬时的；如果能把结果持续保存为表格和图表，价值就变成了长期积累。这个项目让玩家可以把分析结果沉淀下来，用于：

- 长期复盘
- 趋势观察
- 多模式比较
- 数据归档
- 举报或举证时保留截图依据

### 3. 它体现了一个小项目如何从脚本成长为工具

这个项目的意义还在于它很典型：一开始只是快速打通功能，但随着真实使用中不断遇到问题，逐步补上了稳定性、配置化、输出能力、结构重构和可视化。它展示了一个个人项目如何从“能跑”演化成“可重复使用”。

### 4. 它解决的是现实摩擦，不是纸面需求

很多项目只是在功能上看起来完整，但没有真正处理现实环境的复杂性。这个项目的价值恰恰在于，它认真处理了最脏、最烦、最不稳定的部分，比如反爬验证、提交节流、失败恢复和重复任务跳过。

这类工作也许不华丽，但最有实际价值。

## 总结

`Batch Mortal` 的核心价值，不只是“批量分析牌谱”，而是把一个零散、重复、容易出错的人工流程，做成了一套相对稳定、可复用、可积累结果的自动化工具。

它的亮点不在花哨，而在务实：

- 功能链路完整
- 真实问题处理充分
- 配置和结果沉淀意识明确
- 已经具备一定工程化形态

如果继续发展，这个项目完全可以从个人使用脚本，进一步演化成一个更成熟的分析工具。
